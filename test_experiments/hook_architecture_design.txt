================================================================================
增强钩子架构设计
================================================================================

核心思想：在官方函数的关键位置插入钩子，而不是替换整个函数

官方函数增强点分析
--------------------------------------------------------------------------------

函数: check_toc
  描述: 检测文档是否有目录
  当前实现: 文本扫描前20页
  弱点: 可能漏掉不寻常格式的TOC
  钩子名: on_check_toc
  增强方式: VLM缩略图验证TOC检测结果

函数: toc_extractor
  描述: 从TOC页提取条目
  当前实现: LLM从文本提取
  弱点: 可能提取错误或遗漏
  钩子名: on_toc_extracted
  增强方式: VLM视觉验证提取的条目

函数: calculate_page_offset
  描述: 计算页码偏移量
  当前实现: 文本匹配计算中位数
  弱点: 文本提取错误导致计算错误
  钩子名: on_offset_calculated
  增强方式: VLM验证offset是否正确

函数: process_no_toc
  描述: 无目录文档生成结构
  当前实现: LLM从文本生成
  弱点: 可能漏掉视觉分隔页
  钩子名: on_structure_generated
  增强方式: VLM分析分隔页和整体布局

函数: verify_toc
  描述: 验证TOC准确率
  当前实现: LLM文本匹配验证
  弱点: 文本质量差时验证失败
  钩子名: on_verify
  增强方式: VLM视觉验证替代文本验证

函数: fix_incorrect_toc
  描述: 修正错误的TOC条目
  当前实现: 在相邻正确条目间搜索
  弱点: 搜索范围可能不够
  钩子名: on_fix_incorrect
  增强方式: VLM视觉定位正确页码

钩子接口设计
================================================================================

class EnhancementHooks:
    """多模态增强钩子 - 在官方流程关键节点提供增强能力"""

    # 钩子1: TOC检测增强
    async def on_check_toc(self, page_list, check_toc_result):
        """
        在check_toc后调用
        
        Args:
            page_list: 页面文本列表
            check_toc_result: 官方检测结果
            
        Returns:
            修正后的check_toc_result，或None（使用官方结果）
        """
        pass

    # 钩子2: TOC提取增强
    async def on_toc_extracted(self, toc_items, page_list):
        """
        在TOC提取后调用
        
        Args:
            toc_items: 提取的TOC条目列表
            page_list: 页面文本列表
            
        Returns:
            修正后的toc_items，或None（使用官方结果）
        """
        pass

    # 钩子3: 偏移量计算增强
    async def on_offset_calculated(self, offset, toc_items, page_list):
        """
        在offset计算后调用
        
        Args:
            offset: 计算出的偏移量
            toc_items: TOC条目
            page_list: 页面文本列表
            
        Returns:
            修正后的offset，或None（使用官方结果）
        """
        pass

    # 钩子4: 结构生成增强
    async def on_structure_generated(self, structure, page_list, analysis_info):
        """
        在结构生成后调用（process_no_toc模式）
        
        Args:
            structure: 生成的结构列表
            page_list: 页面文本列表
            analysis_info: 文档分析信息（包含分隔页等）
            
        Returns:
            修正后的structure，或None（使用官方结果）
        """
        pass

    # 钩子5: 验证增强
    async def on_verify(self, accuracy, incorrect_items, page_list):
        """
        在verify_toc后调用
        
        Args:
            accuracy: 验证准确率
            incorrect_items: 验证错误的条目
            page_list: 页面文本列表
            
        Returns:
            (修正后的accuracy, 修正后的incorrect_items)，或None
        """
        pass

    # 钩子6: 错误修正增强
    async def on_fix_incorrect(self, incorrect_items, page_list, neighbor_range):
        """
        在fix_incorrect_toc_with_retries前调用
        
        Args:
            incorrect_items: 需要修正的条目

            neighbor_range: 相邻正确条目的范围
            
        Returns:
            修正后的incorrect_items，或None（使用官方修正）
        """
        pass

使用方式
================================================================================

# 1. 创建自定义增强钩子
class MultimodalEnhancementHooks(EnhancementHooks):
    async def on_check_toc(self, page_list, check_toc_result):
        # 如果官方未检测到TOC，用VLM确认
        if not check_toc_result.get('toc_content'):
            vlm_result = await vlm_detect_toc(page_list)
            if vlm_result:
                return vlm_result
        return None  # 使用官方结果

    async def on_structure_generated(self, structure, page_list, analysis_info):
        # 如果有分隔页，用VLM分析
        if analysis_info.get('has_dividers'):
            vlm_structure = await vlm_analyze_dividers(page_list, analysis_info['dividers'])
            if vlm_structure:
                return vlm_structure
        return None

# 2. 在官方流程中使用
hooks = MultimodalEnhancementHooks()
result = await tree_parser(page_list, opt, hooks=hooks)

# 3. 官方tree_parser内部逻辑（伪代码）
async def tree_parser(page_list, opt, hooks=None):
    # 步骤1: 检测TOC
    check_toc_result = check_toc(page_list, opt)
    
    # 钩子: TOC检测增强
    if hooks:
        enhanced_result = await hooks.on_check_toc(page_list, check_toc_result)
        if enhanced_result is not None:
            check_toc_result = enhanced_result
    
    # 步骤2: 根据检测结果路由
    if check_toc_result.get('toc_content'):
        toc_items = await meta_processor(...)
    else:
        toc_items = await meta_processor(mode='process_no_toc', ...)
    
    # 步骤3: 验证
    accuracy, incorrect = await verify_toc(page_list, toc_items)
    
    # 钩子: 验证增强
    if hooks and accuracy < 0.8:
        enhanced_verify = await hooks.on_verify(accuracy, incorrect, page_list)
        if enhanced_verify:
            accuracy, incorrect = enhanced_verify
    
    # 步骤4: 修正错误
    if accuracy < 0.6:
        # 钩子: 错误修正增强
        if hooks:
            enhanced_fix = await hooks.on_fix_incorrect(incorrect, page_list, ...)
            if enhanced_fix:
                incorrect = enhanced_fix
        
        # 官方修正逻辑
        fixed_items = await fix_incorrect_toc_with_retries(...)
    
    return toc_items

================================================================================
优势分析
================================================================================

vs 简单触发方案:
  [OK] 更精细的增强粒度（在每个函数内部）
  [OK] 保持官方API不变
  [OK] 模块化，每个钩子独立
  [OK] 失败自动回退到官方逻辑

vs 替换官方方案:
  [OK] 保留官方核心逻辑
  [OK] 渐进式增强
  [OK] 风险可控
  [OK] 易于回滚

vs 我们当前的Branch A/B/C:
  [OK] 没有并行路径
  [OK] 没有复杂的条件分支
  [OK] 逻辑清晰：官方为主，钩子增强
  [OK] 易于测试（每个钩子独立测试）

================================================================================
实施建议
================================================================================

Phase 1: 基础框架（1-2天）
  1. 创建EnhancementHooks基类
  2. 修改tree_parser接受hooks参数
  3. 在关键位置插入钩子调用
  4. 测试无hooks时的行为（应与官方一致）

Phase 2: 实现关键钩子（2-3天）
  1. on_structure_generated: 分隔页分析
  2. on_check_toc: TOC检测增强
  3. on_verify: 视觉验证
  4. 每个钩子独立测试

Phase 3: 集成测试（1-2天）
  1. 在24个文档上测试
  2. 对比增强前后的准确率
  3. 测量性能和成本

Phase 4: 优化（持续）
  1. 根据测试结果调整钩子策略
  2. 添加配置项控制钩子启用/禁用
  3. 监控钩子触发率和效果

================================================================================
总结
================================================================================

核心创新：钩子架构
  - 不替换官方流程
  - 在关键节点提供增强能力
  - 保持向后兼容
  - 模块化、可测试、可维护

预期效果:
  - 准确率: ~85% -> ~95%
  - 成本: 增加20-30%（仅对需要增强的文档）
  - 复杂度: 远低于当前Branch A/B/C方案
  - 维护性: 每个钩子独立，易于调试
